# 图片编辑实测清单

更新时间：2026-04-28

这份清单用于第二轮上线前后的人工验证，重点覆盖当前线上图片编辑工作流的 3 个模型、6 条生图线路。

注意：

- 这份清单基于你当前线上返回过的 `/api/image-routes/catalog` 和 `/api/image-models/catalog`。
- 如果上线前线上线路又改过，先重新拉一次 catalog，再按最新线路修正这份清单。
- 视频不在这次测试范围里。

## 一、测试目标

确认下面这几件事同时成立：

1. 图片编辑入口统一走画布编辑面板，不再弹旧重绘窗口。
2. 编辑时是按 `modelId + routeId` 精确提交，不会串到默认线路。
3. 三个模型的编辑结果都能：
   - 正常返图
   - 正常显示在画布
   - 正常写入历史记录
   - 正常上传 OSS
   - 支持继续编辑
   - 支持“应用到原图”

## 二、测试前准备

上线前先确认：

1. 线上 `.env`
   - `GENERATED_ASSET_STORAGE=on`
2. OSS CORS 已配置
3. 已登录一个有点数的测试账号
4. 准备 3 张测试图
   - 人像图
   - 商品/物体图
   - 背景复杂的场景图

建议统一用这 3 类编辑动作测：

1. 局部替换
2. 擦除物体
3. 修复细节

## 三、线路清单

### 1. Nano Banana Pro

#### 1.1 线路一

- `modelId`: `nano-banana`
- `routeId`: `nano-banana-pro`
- `line`: `线路一`
- `mode`: `async`

测试项：

1. 选中图片，进入“图片编辑”
2. 涂抹人物手部区域，选择“修复细节”
3. 提交后应出现占位节点
4. 轮询完成后，右侧生成编辑结果
5. 右键结果图，点击“应用到原图”
6. 原图内容应更新，且历史记录保留

重点看：

- 异步任务是否稳定回填
- 应用到原图后是否保留可继续编辑状态

#### 1.2 线路二

- `modelId`: `nano-banana`
- `routeId`: `nano-banana-pro-线路二`
- `line`: `线路二`
- `mode`: `async`

测试项：

1. 用场景图做“换背景”
2. 观察点数显示是否和当前线路一致
3. 生成结果后，继续对结果图再做一次“局部替换”

重点看：

- 第二次继续编辑是否仍走选中的线路二
- 不要回落到默认线路

#### 1.3 线路四

- `modelId`: `nano-banana`
- `routeId`: `nano-banana-pro-line4`
- `line`: `线路四`
- `mode`: `sync`

测试项：

1. 用商品图做“局部替换”
2. 提交后应较快同步返图
3. 结果图在画布中必须直接显示
4. 历史记录中能打开
5. 图片地址应为 OSS 地址

重点看：

- 同步线路是否直接回填成功
- 画布不应再出现一直 loading 的问题
- OSS 地址应可直接访问

### 2. Nano Banana 2

#### 2.1 线路一

- `modelId`: `nano-banana-2`
- `routeId`: `nano-banana-2-线路一`
- `line`: `线路一`
- `mode`: `async`

说明：

- 当前线上线路一可能同时存在 `nano-banana-2` 和 `nano-banana-2-线路一` 两条相近配置。
- 实测时优先选择编辑面板里展示为可编辑、描述最完整的那条。
- 如果两条都显示，要分别记录它们的 `routeId`，确认没有串线。

测试项：

1. 用人像图做“局部替换”
2. 用创意提示词替换头饰、服装或背景元素
3. 生成后继续编辑一次

重点看：

- 创意替换是否明显优于简单修补
- 继续编辑时比例是否保持原图比例

#### 2.2 线路三

- `modelId`: `nano-banana-2`
- `routeId`: `nano-banana-2-线路三`
- `line`: `线路三`
- `mode`: `sync`

测试项：

1. 用场景图做“擦除物体”
2. 再测一次“风格调整”
3. 检查同步返回时，右侧是否生成新节点

重点看：

- 同步结果能否稳定显示
- 风格调整是否保持大构图不乱

### 3. GPT-image-2

#### 3.1 线路一

- `modelId`: `gpt-image-2`
- `routeId`: `gpt-image-2`
- `line`: `线路一`
- `mode`: `async`

测试项：

1. 用背景复杂图做“擦除物体”
2. 再测“修复细节”
3. 分别测试 `1k`、`2k`

重点看：

- GPT-image-2 的尺寸映射是否正常
- 小改图是否明显更快、更便宜

## 四、每条线路通用检查项

每条线路都要检查这 8 项：

1. 入口文案是否统一为“图片编辑”
2. 不再出现旧重绘弹窗
3. 默认比例是否跟随原图
4. 提交后结果是否生成在原图右侧
5. 历史记录是否写入成功
6. 图片是否上传到 OSS
7. 右键“应用到原图”是否可用
8. 应用后是否还能继续编辑

## 五、异常检查

如果有问题，优先记录这几个维度：

1. `modelId`
2. `routeId`
3. 编辑类型
4. 原图比例
5. 尺寸
6. 是同步还是异步
7. 画布异常 / 历史异常 / OSS 异常 / 点数异常

建议用这个格式记录：

```text
时间：
账号：
modelId：
routeId：
编辑类型：
尺寸：
比例：
问题现象：
控制台报错：
服务端日志：
是否影响历史记录：
是否影响 OSS：
```

## 六、上线通过标准

满足下面条件，就可以认为这版图片编辑可上线：

1. 六条线路都能成功出至少一张编辑图
2. 同步线路和异步线路都能正常回填画布
3. 历史记录和 OSS 全部正常
4. “应用到原图”至少在 6 条线路里都测通 1 次
5. 没有再出现旧重绘窗口和旧文案
